# 低雷諾數與大展弦比人力飛機之 AVL 3.40 批次自動化與氣動導數解析深度報告

在現代人力飛機（Human-Powered Aircraft, HPA）的設計與開發中，極高的展弦比（High Aspect Ratio）與極低的雷諾數（Low Reynolds Number，通常落於 $5 \times 10^5$ 區間）是其最顯著的空氣動力學特徵 \[1, 2\]。這類飛行器為了將誘導阻力降至最低，往往具備極端細長的機翼，同時為配合人力極為有限的輸出功率，整體結構必須極度輕量化。這種氣動與結構特性的結合，導致飛行器的縱向與橫向動態穩定性邊界極為狹窄，且極易受到微小氣動擾動的影響 \[3, 4\]。因此，在進行多學科設計最佳化（Multidisciplinary Design Optimization, MDO）流程時，必須依賴高效且精確的氣動力求解引擎來頻繁評估不同幾何構型下的穩定性導數（Stability Derivatives）與配平（Trim）狀態。

由麻省理工學院（MIT）Mark Drela 教授與 Harold Youngren 開發的 Athena Vortex Lattice (AVL 3.40) 軟體，憑藉其基於渦格法（Vortex Lattice Method, VLM）的底層架構，成為處理此類問題的工程標準 \[5, 6\]。AVL 假設流場為無黏性、不可壓縮且處於準定常狀態（Quasi-steady），能以極低的計算成本快速求解複雜幾何構型的氣動力係數、控制面導數與特徵模態 \[6, 7\]。然而，在自動化 MDO 流程中，工程演算法必須無人值守地頻繁呼叫 AVL，並從其生成的 `.st` 檔案中精確萃取 Jacobian 矩陣參數 \[6, 8\]。此過程涉及複雜的底層 CLI 通訊、嚴格的幾何控制面映射邏輯，以及針對早期 Fortran 程式特有輸出格式的文字解析技術。本文將針對 AVL 3.40 的批次執行架構、多控制面陣列的映射關係，以及在自動化過程中極易觸發的系統陷阱進行全面且深度的工程剖析。

## 關鍵字速查與物理意義解析

## 最小工作範例（Minimum Working Example, MWE）

為了在 Python 或 Bash 等外部環境中穩定驅動 AVL 進行多學科最佳化，必須建立標準化的幾何描述模板、精確的命令列控制腳本，以及具備容錯能力的數據解析器。以下三個相互關聯的工程代碼片段，完整展示了從模型定義到導數萃取的自動化生命週期。

### 片段一：具備複雜多控制面映射的 `.avl` 幾何模板

在人力飛機的氣動設計中，主翼通常具備控制升力分佈的襟翼（Flap）與控制滾轉的副翼（Aileron），而尾翼則具備升降舵（Elevator）與方向舵（Rudder）\[9, 15\]。在 AVL 的解析邏輯中，控制面的內部代號（`d1`, `d2`, `d3`...）完全取決於 `CONTROL` 關鍵字在 `.avl` 檔案中被讀取的先後順序。任何透過腳本自動生成的幾何，都必須嚴格鎖定此順序，否則將導致配平約束綁定錯誤的物理控制面 \[14\]。

針對上述模型，AVL 會在記憶體中建立一個控制向量映射表：`d1` 映射並控制全機的 `flap`，`d2` 映射 `aileron`，`d3` 映射 `elevator` \[9\]。在 MDO 過程中，如果為了測試擾流板（Spoiler）而將其定義插在 `flap` 與 `aileron` 之間，則 `d2` 會立刻變成擾流板，原本的 `d3` 會變成副翼。這項特性要求開發者在自動化腳本中，必須具備動態解析對應關係的能力，絕對不可在最佳化腳本中寫死變數編號 \[14, 16\]。

### 片段二：Command Mode 下的穩定配平指令序列

為了獲取真實的靜態穩定裕度，必須讓飛機在特定的目標升力係數（$C_L$）下進行配平。這意味著必須釋放攻角（Alpha）與升降舵（`d3`）作為自由變數，同時強制系統的淨俯仰力矩（Pitching Moment）為零 \[17\]。這項操作必須轉換為一系列能精確驅動 AVL 的文字指令流，任何一個空行的遺漏都會導致選單層級錯亂。

```bash
# trim_sequence.run
LOAD HPA_Optimized_Wing_Tail.avl
MASS HPA_Mass_Distribution.mass
MSET 0          # 極為關鍵：強制將力矩參考點移動至質量檔案計算出的真實重心 (CG)
OPER            # 進入操作與配平選單
M               # 進入約束與自由度修改子選單
# --- 約束設定 ---
C               # 選擇升力係數約束
C 0.95          # 目標 CL = 0.95 (典型人力飛機巡航 CL)
A               # 宣告攻角 (Alpha) 為自由變數以滿足上述 CL
PM 0.0          # 選擇俯仰力矩約束，目標值設為 0.0
D3              # 宣告 d3 (Elevator) 為自由變數以滿足俯仰力矩約束
D1              # 選擇 d1 (Flap)
D1 5.0          # 強制將襟翼角度固定為 5 度
D2              # 選擇 d2 (Aileron)
D2 0.0          # 強制將副翼角度固定為 0 度
# --- 返回與求解 ---
                # (空行) 模擬 Enter 鍵，退出 M 子選單返回 OPER
X               # 執行 Newton-Raphson 求解器進行配平
ST hpa_derivatives.st  # 將配平後的穩定性與控制導數矩陣寫入檔案
                # (空行) 模擬 Enter 鍵，退出 OPER 選單
QUIT            # 正常終止 AVL 進程
```

上述指令序列中的 `MSET 0` 是一道關鍵的防線 \[2\]。在設計優化過程中，結構重量分佈不斷改變，若未執行 `MSET 0`，AVL 會繼續使用 `.avl` 標頭中靜態的 `Xref` 計算力矩，這將導致配平所需的升降舵角度產生非物理的巨大偏差 \[10\]。

### 片段三：Python 子行程（Subprocess）自動化執行與強固解析器

在 MDO 框架（如 OpenMDAO）中，Python 被廣泛用作流程控制器。利用 `subprocess.Popen` 來驅動 AVL 時，最常遭遇的問題是進程死鎖（Deadlock）以及 Fortran 檔案的空白截斷錯誤 \[18, 19\]。下方的 Python 腳本展示了如何安全地發送指令流，並利用正規表達式（Regular Expression）避開 Fortran 文字輸出的排版陷阱。

```python
import subprocess
import os
import re

def execute_avl_and_extract_derivatives(avl_exe_path, run_script_path, output_st_path):
    """
    透過標準輸入流驅動 AVL，並穩健地解析.st 穩定性導數檔案。
    """
    # 確保先前的殘留檔案被清除，避免 AVL 停留在 "Overwrite? (Y/N)" 的阻塞狀態
    if os.path.exists(output_st_path):
        os.remove(output_st_path)

    # 讀取預先準備好的指令序列 (包含 LOAD, OPER, X, ST, QUIT)
    with open(run_script_path, 'r') as file:
        command_sequence = file.read()

    # 使用 subprocess 啟動 AVL
    # 設置 text=True 允許直接傳遞與接收字串，避免 bytes 編碼問題 [18]
    try:
        process = subprocess.Popen(
            [avl_exe_path],
            stdin=subprocess.PIPE,
            stdout=subprocess.PIPE,
            stderr=subprocess.PIPE,
            text=True
        )
        
        # 使用 communicate 發送指令流。此方法會自動發送 EOF 並等待進程結束，
        # 是避免底層作業系統 Pipe Buffer 填滿導致 Deadlock 的唯一安全解法 [18]。
        stdout, stderr = process.communicate(input=command_sequence)
        
        if process.returncode!= 0:
            raise RuntimeError(f"AVL 執行異常終止。返回碼: {process.returncode}")
            
    except Exception as e:
        print(f"[Error] AVL 進程啟動或通訊失敗: {e}")
        return None

    # ==========================================
    # 執行安全且強固的.st 檔案解析
    # ==========================================
    extracted_data = {}
    if not os.path.exists(output_st_path):
        print("[Error] 找不到輸出的.st 檔案，可能配平失敗或發生發散。")
        return extracted_data

    with open(output_st_path, 'r') as f:
        content = f.read()

        # [地雷迴避] Fortran 格式化陷阱處理
        # 由於當負號出現時，Fortran 的輸出會吃掉空白 (例如 0.12345-0.98765) [19]
        # 絕對不能依賴 string.split()。此處利用正規表達式捕捉變數名稱與浮點數的配對。
        # 正規表達式邏輯：捕捉英數底線組成的變數名，略過等號與空白，捕捉可帶負號的浮點數。
        pattern = re.compile(r'(?P<var_name>[A-Za-z0-9_]+)\s*=\s*(?P<value>-?\d+\.\d+)')
        
        for match in pattern.finditer(content):
            var_name = match.group('var_name')
            value = float(match.group('value'))
            extracted_data[var_name] = value

    return extracted_data

# 使用範例
# aero_derivatives = execute_avl_and_extract_derivatives('./avl', 'trim_sequence.run', 'hpa.st')
# if aero_derivatives:
#     print(f"Pitch Stiffness (Cma): {aero_derivatives.get('Cma', 'N/A')} 1/rad")
#     print(f"Elevator Control Power (Cmd3): {aero_derivatives.get('Cmd3', 'N/A')} 1/deg")
```

這套腳本具備高度的工程價值，因為它直接繞過了傳統字串分割法在處理緊湊資料陣列時的致命缺陷，確保每次 MDO 疊代都能正確讀取到目標變數 \[19\]。

## 常見地雷（Core Pitfalls in Automation）

將 AVL 此類具備強烈互動性質的傳統程式嵌入現代黑盒子（Black-box）最佳化管線時，系統極容易在邊界條件下崩潰。根據深度工程實踐，以下三個陷阱是導致 MDO 流程中斷的主因，必須在演算法架構設計階段予以排除。

### 地雷一：多控制面導致 Jacobian 矩陣奇異（Singular Jacobian Matrix）與配平發散

**現象描述：**
當 MDO 腳本進入 `OPER` 模組並發送 `X`（Execute）指令時，AVL 在標準輸出中印出「Singular Jacobian Matrix」或「Matrix is rank deficient」的錯誤警告，導致求解器無法找到配平狀態，`.st` 檔案生成失敗或數據為空 \[16, 20\]。

**物理與數學機制：**
AVL 的配平系統本質上是一個牛頓-拉弗森（Newton-Raphson）非線性根尋找器。它透過微小擾動來計算靈敏度矩陣（Jacobian Matrix），藉以反演尋找滿足約束條件的控制變數。當矩陣呈現奇異（Singular）或條件數極差（Ill-conditioned）時，意味著數學系統處於欠定（Under-determined）或控制變數間存在強烈的線性相依性 \[16, 21\]。

在人力飛機中，為了減少配平阻力，設計者可能配置了具有冗餘控制能力的機構，例如「全動式平尾（All-moving tail）」加上「平尾後緣升降舵（Elevator）」。如果在自動化腳本中，同時將全動平尾（`D3`）與升降舵（`D4`）釋放為自由變數，要求系統去滿足單一的俯仰力矩約束（`PM = 0`），這兩個變數對俯仰力矩的貢獻呈現高度線性重疊。AVL 無法決定該分配多少角度給平尾、多少給升降舵，導致 Jacobian 矩陣某兩列的線性獨立性喪失，行列式趨近於零，進而使求解矩陣無法反轉而崩潰 \[17, 22\]。

另一個常見的奇異點陷阱是試圖在完全對稱的飛機上，釋放副翼（`d2`）去配平滾轉力矩（`RM = 0`）。由於幾何對稱，滾轉力矩自然為零，副翼偏折產生的靈敏度導數在無側滑的條件下幾乎失去參考價值，使得求解器除以趨近於零的梯度而發散 \[16, 23\]。

**迴避與解決策略：**
在任何配平指令序列中，必須嚴格遵守「控制自由度數量等於約束條件數量」的鐵則。針對具備冗餘控制面的飛行器，應將額外的控制面（如襟翼或全動平尾的角度）由外層的 MDO 演算法作為給定參數（例如寫死 `D1 5.0`），而僅保留一個最有效的控制面（如 `D4`）留給 AVL 的內部求解器進行配平 \[17\]。

### 地雷二：`.st` 檔案在不同作業系統下的空白符號解析陷阱（Whitespace Parsing Trap）

**現象描述：**
最佳化流程在執行了數百次迭代後，Python 腳本突然拋出 `IndexError` 或是浮點數轉換 `ValueError`。檢查輸出的 `.st` 檔案時，發現穩定性矩陣的數值連在一起，破壞了原有的欄位結構 \[19, 24\]。

**物理與數學機制：**
這是早期基於 Fortran 的工程軟體最臭名昭著的 I/O 問題之一。AVL 匯出 `.st` 檔案時，使用的是基於字元寬度的嚴格格式化輸出（例如 `FORMAT(6F10.5)`，意即每個數字佔 10 個字元，小數點後 5 位）。在正常的氣動區間內，數字可能顯示為：
`   0.04321    0.01234   -0.00567`
然而，在 MDO 過程中，演算法經常會探索極端的幾何設計。當某個氣動導數的數值變得極大或由正轉負時，負號 `-` 或者多出來的整數位數，會無情地吞噬掉原本預留作為欄位分隔的空白字元（Whitespace）\[19\]。例如上述數據可能異變為：
`   0.04321-12.01234   -0.00567`
若自動化腳本依賴 Python 預設的 `string.split()` 函數（該函數以空白作為切分基準），它會將 `0.04321-12.01234` 識別為單一的無效字串，從而導致整個矩陣的列數偏移，破壞後續 `d1, d2, d3` 所有控制導數矩陣位置的萃取邏輯。此外，在跨平台（Windows 與 Linux）環境下，換行符號（CRLF 與 LF）的差異也會干擾粗劣的逐行解析器 \[24, 25\]。

**迴避與解決策略：**
徹底揚棄以空白字元為基礎的字串分割法。應採用固定寬度切片（Fixed-width Slicing）或是更具彈性的正規表達式（Regular Expression）來處理數值萃取 \[19\]。如前述的 Python 最小工作範例所示，利用 `re.finditer(r'(?P<var_name>[A-Za-z0-9_]+)\s*=\s*(?P<value>-?\d+\.\d+)')` 能夠精確且無視排版崩壞地抓取任何出現的變數與其對應數值，這是確保 MDO 腳本具備工業級穩定度的關鍵技術。

### 地雷三：參考點 (Xref, Yref, Zref) 設錯導致的力矩與穩定度災難

**現象描述：**
在最佳化過程中，AVL 輸出的配平攻角或升降舵偏角出現不合理的極端值（如配平攻角高達 30 度），同時 `.st` 檔案中的縱向靜穩定度指標 $C_{m_\alpha}$ 顯示飛機處於極度不穩定的發散狀態，即使幾何外觀與尾容量（Tail Volume）設計完全合理 \[5, 10\]。

**物理與數學機制：**
所有氣動力矩（包括俯仰、滾轉與偏航力矩）以及其對應的穩定性導數，都必須依附於一個特定的空間參考點進行積分與計算。在真實飛行中，這個參考點是飛機的重心（Center of Gravity, CG）\[13, 15\]。在 AVL 中，預設的力矩參考點是由 `.avl` 檔案標頭的 `Xref, Yref, Zref` 參數定義的 \[10\]。

在動態的 MDO 流程中，隨著機翼面積的增減、駕駛艙位置的前後移動，整架人力飛機的實際重心會不斷發生位移。如果自動化腳本只負責重寫 `.avl` 檔案以改變氣動外型，卻沒有動態計算並更新 `.avl` 檔案標頭的 `Xref`，或者沒有將獨立計算好的 `.mass` 檔案重心引入求解器，AVL 就會繼續相對於一個錯誤的、靜態的幾何原點來計算俯仰力矩 \[15\]。對於人力飛機這種展長可能達到 30 公尺、弦長極大的飛行器而言，幾公分的力臂誤差就會被巨大的機翼升力極度放大。這會產生虛假的低頭或抬頭力矩，迫使牛頓求解器給出極端且荒謬的升降舵配平角度，進一步產生極大的配平阻力（Trim Drag），徹底摧毀最佳化目標函數的可信度 \[2, 26\]。

**迴避與解決策略：**
建構嚴謹的參考點連動機制。在每次進行氣動分析前，MDO 結構模組必須先計算出當前構型的精確質量分佈，並輸出為 AVL 可讀的 `.mass` 檔案 \[6\]。在 CLI 命令序列進入 `OPER` 之前，務必強制寫入 `MSET 0` 指令 \[2, 6\]。此指令會覆蓋 `.avl` 標頭的靜態定義，強迫 AVL 將力矩積分的原點平移至由 `.mass` 檔案物理積分出的真實重心位置，從根本上確保了 $C_{m_\alpha}$ 與控制導數計算的絕對物理正確性 \[5, 26\]。此外，在解析 `.st` 檔的導數時，須牢記攻角與側滑角的導數（如 $C_{m_\alpha}$）預設單位為 `1/rad`，而控制面的導數（如 $C_{m_{\delta e}}$ 即 $Cmd3$）常因內部設定被輸出為 `1/deg`，維度轉換的疏忽將導致控制律設計完全失效 \[27\]。

綜合上述分析，將 AVL 3.40 整合至人力飛機的多學科設計最佳化流程，絕非單純的腳本串接，而是一項需要深度融合渦格氣動力學理論、非線性數值求解邊界，以及軟體工程容錯機制的系統性工程。只有嚴密控制幾何映射序列、精確定義配平約束，並具備防禦性的資料解析策略，才能在無數次的自動化迭代中，榨取出這套經典氣動引擎的極致效能。
