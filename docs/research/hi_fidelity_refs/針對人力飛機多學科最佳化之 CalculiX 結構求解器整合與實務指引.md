# 針對人力飛機多學科最佳化之 CalculiX 結構求解器整合與實務指引

人力飛機（Human-Powered Aircraft, HPA）的設計與開發，被視為航空工程領域中極致輕量化與極高空氣動力效率的終極挑戰。在此類飛行器的設計過程中，高展弦比（High Aspect Ratio）機翼是降低誘導阻力的必然幾何選擇，然而這將導致機翼主結構（通常為薄壁複合材料管件）承受極大的彎矩與扭矩。為同時滿足起飛重量限制與飛行中的氣動彈性穩定性，導入多學科設計最佳化（Multidisciplinary Design Optimization, MDO）框架是當代研發的標準作業流程。在此框架中，計算固體力學（Computational Solid Mechanics）求解器的精準度、自動化相容性與運算效率，直接決定了最佳化演算法（如梯度下降法或遺傳演算法）能否在有限的運算資源內收斂至全局最佳解。

開源有限元素分析（Finite Element Analysis, FEA）軟體 CalculiX 2.22 憑藉其與商業軟體 Abaqus 高度相似的輸入語法、強大的非線性求解能力以及無授權限制的平行運算特性，成為串接 Python 自動化 MDO 框架的理想結構求解器。本報告針對 HPA 專案的特殊需求，深度剖析 CalculiX 2.22 在處理薄壁管件與碳纖維複合材料（CFRP）時的底層數學運作邏輯與工程實務。報告內容涵蓋關鍵語法字典的建立、針對複合材料主樑的最小工作範例（Minimum Working Examples, MWE），以及深度剖析與網格生成器（如 Gmsh）對接時的致命陷阱，藉此為航太結構工程團隊提供一份可直接實作且具備高度防呆機制的技術指引。

## 關鍵字速查與底層力學機制解析

在建構 Python 自動化腳本以生成 CalculiX 輸入檔（`.inp`）的過程中，精準掌握關鍵字的階層關係與語法規範是確保求解器順利初始化的首要任務。CalculiX 的輸入檔結構嚴格區分為「模型定義區（Model Definition）」與「歷程定義區（History Definition）」，兩者透過特定關鍵字進行隔離與切換。下表整理了針對 HPA 結構分析最核心的 20 個關鍵字，並提供其實作面與演算法層面的技術定義。

| 關鍵字標籤 | 模組分類 | 實作面解釋與自動化框架注入邏輯 | 
|---|---|---|
| `*STEP` | 歷程控制 | 定義分析階段的起點，扮演分隔模型幾何定義區與邊界負載歷程區的絕對界線，所有外力變化皆須包覆於此關鍵字之下 \[1\]。 | 
| `*STATIC` | 求解器宣告 | 啟動隱式靜力學求解矩陣，接受時間增量參數以控制牛頓-拉弗森（Newton-Raphson）迭代，涉及大變形時需附加 `NLGEOM` 參數更新剛度矩陣。 | 
| `*BUCKLE` | 求解器宣告 | 啟動線性特徵值挫曲分析，後接四個參數：提取的模態數、精度、Lanczos 向量數與最大迭代次數，為薄壁管評估穩定性的核心 \[2, 3\]。 | 
| `*BOUNDARY` | 邊界條件 | 定義節點位移與旋轉自由度（DOF 1-6）的拘束狀態，Python 框架常將翼根節點集合的 DOF 1-6 設為零以完美模擬懸臂樑邊界。 | 
| `*CLOAD` | 負載施加 | 於單一節點或節點集合上施加集中力或集中力矩，實務上常結合剛體虛擬節點以等效施加全翼段的空氣動力學俯仰力矩 \[4\]。 | 
| `*DLOAD` | 負載施加 | 定義表面分布載荷或全域慣性載荷，實作上最常用於管內外壓力差定義，或透過 `GRAV` 參數結合重力加速度向量施加結構自重 \[5, 6\]。 | 
| `*NODE FILE` | 結果輸出 | 指定輸出至二進位結果檔（`.frd`）的節點層級變數，如位移場（`U`）、反作用力（`RF`），是 Python 後處理器提取最佳化目標函數的接口 \[7\]。 | 
| `*EL FILE` | 結果輸出 | 指定輸出至 `.frd` 的元素積分點變數，實務上必填應力（`S`）與應變（`E`）張量，用於評估碳纖維疊層的蔡-吳（Tsai-Wu）破壞準則。 | 
| `*NSET` | 幾何群組 | 定義節點集合，自動化流程中需將 Gmsh 輸出的實體群組精確映射為 NSET 名稱，作為 `*BOUNDARY` 或運動學綁定的施加目標 \[8\]。 | 
| `*ELSET` | 幾何群組 | 定義元素集合，在 `.inp` 檔案中扮演將純幾何網格與具體材料屬性（如厚度、疊層角度）強耦合綁定的資料結構橋樑。 | 
| `*MATERIAL` | 材料模型 | 宣告材料物理定義的起始區塊，接受 `NAME` 字串參數，其後必須緊接描述彈性、塑性或密度等物理定律的關聯關鍵字。 | 
| `*ELASTIC` | 材料屬性 | 定義彈性常數，對於 HPA 碳纖維主樑需附加 `TYPE=ORTHO` 參數，以依序輸入正交非等向性材料的三維工程常數及剪切模數。 | 
| `*DENSITY` | 材料屬性 | 定義質量密度，若模型缺乏此項定義，`*DLOAD, GRAV` 將無法組裝全域質量矩陣，導致重力分析直接報錯或失效 \[6\]。 | 
| `*SHELL SECTION` | 元素截面 | 宣告殼元素物理厚度與對應材料，複合材料需附加 `COMPOSITE` 參數並逐行定義每層厚度、材料及方向名稱，不支援節點厚度輸入 \[9\]。 | 
| `*ORIENTATION` | 局部座標系 | 定義材料主軸局部座標系，在輸入檔中扮演將正交非等向性纖維方向（如 0° 或 45°）精確對齊至機翼主樑幾何主軸的空間映射角色 \[9, 10\]。 | 
| `*EQUATION` | 多點約束 | 建立節點自由度間的線性代數關係，常用於將不同網格密度的翼肋與主樑進行平滑的運動學綁定，需嚴格避免從屬自由度重複定義 \[11\]。 | 
| `*RIGID BODY` | 運動學拘束 | 將指定節點集合剛性綁定至單一虛擬參考節點，實作上等同於 Nastran RBE2 元素，常用於將翼肋周邊節點收斂至一點以傳遞扭矩 \[4\]。 | 
| `*TRANSFORM` | 座標轉換 | 將選定節點的自由度從全域直角座標轉換至指定的局部座標系，專門用於處理具備上反角（Dihedral）或後掠角結構的局部邊界拘束 \[12, 13\]。 | 
| `*AMPLITUDE` | 負載控制 | 定義隨時間變化的載荷乘數曲線，在單一分析步內若需模擬從靜止到極限負載的漸進式非線性加載，需將此關鍵字與 `*CLOAD` 綁定 \[14\]。 | 
| `*INCLUDE` | 模組化載入 | 動態讀取外部純文字檔，Python 框架應利用此指令將龐大的網格連通性資料獨立抽取，以維持主控 `.inp` 檔案的高可讀性與修改彈性 \[15\]。 | 

上述關鍵字的相互作用構成了 CalculiX 求解結構矩陣的底層邏輯。在線性靜力分析中，`*STATIC` 會指示求解器組裝全域剛度矩陣與負載向量，透過直接法（如 SPOOLES 或 PARDISO 求解器）求解線性代數方程組 \[1\]。對於人力飛機極度細長的機翼結構，幾何非線性效應顯著，此時必須在 `*STATIC` 關鍵字後綴 `NLGEOM` 參數。一旦啟用此參數，求解器將把應變度量從工程應變切換為格林-拉格朗日應變（Green-Lagrange Strain），並在每個時間增量步內迭代更新結構的剛度矩陣，以捕捉大變形下的應力剛化效應（Stress Stiffening）。此外，在定義複合材料時，`*MATERIAL` 與 `*SHELL SECTION` 的耦合極為嚴密。當 `COMPOSITE` 參數被觸發時，求解器會自動切換至經典疊層板理論（Classical Lamination Theory）或一階剪切變形理論（Mindlin-Reissner），這要求 Python 框架在寫入腳本時，必須嚴格確保每一層的厚度總和與模具設計的實體厚度吻合，並透過 `*ORIENTATION` 將局部纖維座標精準投射至網格表面上。

## 最小工作範例：複合材料薄壁管之結構建模

在多學科最佳化迴圈中，結構分析往往被封裝為一個獨立的黑盒子函數。該函數接收空氣動力學模組傳遞的負載分佈以及最佳化演算法指定的幾何參數（如鋪層角度與管壁厚度），隨後生成 `.inp` 檔案並呼叫 CalculiX 執行運算。針對人力飛機中最具代表性的碳纖維薄壁圓管（主樑）結構，以下提供五個涵蓋不同物理行為的最小工作範例（MWE），並深入探討其背後的力學意義與 Python 框架的參數注入策略。

### 場景一：複合材料薄壁管之線性靜力彎曲分析

人力飛機的升力主要分佈於機翼表面，並透過翼肋傳遞至機翼主樑，最終轉化為懸臂樑結構上的強大彎矩。此範例展示如何建構一組正交非等向性材料的碳纖維鋪層（例如 0°/45°/90° 的擬等向性鋪層結構），並對一端完全固定的薄壁管施加翼端向上的等效空氣動力載荷。

在複合材料薄殼力學中，上述 `*SHELL SECTION` 的 `OFFSET=-0.5` 參數設定具有決定性的幾何意義。當網格生成器（如 Gmsh）以機翼主樑的外徑輪廓生成表面網格時，該表面被視為幾何參考面。設定 `-0.5` 將指示 CalculiX 把殼元素的厚度向內側（模具內徑方向）完全偏置 \[9\]。此舉確保了結構外型不會因為疊層厚度的最佳化變更而產生氣動外型的干涉或膨脹。Python MDO 框架在此步驟中，會透過樣板引擎（如 Jinja2）動態替換每一層的 `0.25` 厚度值以及對應的 `ORI_` 方向名稱，藉此搜尋最佳的疊層剛度矩陣（ABD Matrix）組合。

### 場景二：薄壁管之特徵值挫曲分析（Eigenvalue Buckling）

當人力飛機於空中遭遇陣風或進行機動拉升時，機翼會產生極大的上反彎曲變形。在此受力狀態下，薄壁主樑的上半部將承受強烈的軸向壓縮應力。由於複合材料薄壁管的徑厚比通常極大，結構往往在遠低於材料抗壓降伏應力的階段，便發生局部面層皺褶（Face Wrinkling）或全域尤拉挫曲（Euler Buckling）\[18\]。準確預測此臨界挫曲負載是 HPA 結構設計的生命線。

\*BUCKLE

\*NODE FILE

在演算法層面，CalculiX 採用 Lanczos 特徵值求解器來處理挫曲問題。該方法透過在 Krylov 子空間中進行迭代，相較於傳統的子空間迭代法（Subspace Iteration），能更高效地提取大型稀疏矩陣的前幾個最小特徵值。在上述配置中，系統將提取前 5 個挫曲模態形狀及其對應的挫曲乘數（Buckling Factors）。提取的特徵值將直接寫入與任務同名的 `.dat` 報告檔中。Python 框架必須實作一個高效的正則表達式（Regular Expression）解析器，去捕獲 `.dat` 檔內的 `BUCKLING FACTOR` 區塊數值。值得注意的是，若施加的初始參考負載使得結構處於張力狀態，Lanczos 求解器將可能計算出負值的特徵值 \[19\]，這在物理上意味著必須將施加的負載完全反向，結構才會達到挫曲狀態。最佳化演算法的目標函數應設定為將最小正特徵值最大化，並確保其安全係數大於 1.5。

### 場景三：透過剛體約束（Rigid Body）進行氣動彈性扭轉分析

氣動彈性現象（Aeroelasticity）是 HPA 設計不可迴避的難題。不對稱的氣流分佈或翼型的俯仰力矩（Pitching Moment）會對機翼產生強大的扭矩，導致機翼扭轉，進而改變局部攻角（Angle of Attack），引發升力重新分佈甚至氣動發散。在有限元素建模中，將分佈的氣動力矩精確施加於薄壁管的末端邊界是一項挑戰。使用 `*RIGID BODY` 關鍵字可以優雅地解決此負載傳遞問題。

\*HEADING
HPA Wing Spar - Torsional Aeroelastic Loading via Rigid Body
\*INCLUDE, INPUT=mesh_all.inp
\*INCLUDE, INPUT=material_properties.inp

\*\* 建立一個遠離結構的獨立參考節點 (Reference Node) \[4\]
\*\* 為了避免 ID 衝突，將其編號設為極大值
\*NODE
999999, 15000.0, 0.0, 0.0

\*\* 宣告剛體約束，將翼端邊緣的所有節點綁定至該參考節點
\*RIGID BODY, NSET=Tip_Edge_Nodes, REF NODE=999999

\*STEP
\*STATIC
\*BOUNDARY
Root_Nodes, 1, 6, 0.

\*\* 透過參考節點直接施加繞縱軸 (DOF 4) 的空氣動力扭矩
\*CLOAD
999999, 4, 25000.0

\*NODE FILE
U
\*END STEP

採用 `*RIGID BODY` 的數學本質，是在內部建構一個龐大的多點約束（Multipoint Constraint, MPC）矩陣。它強制 `Tip_Edge_Nodes` 集合中的所有節點相對位置保持不變，並嚴格跟隨參考節點 `999999` 的平移與旋轉動作 \[4\]。這個手法極大地簡化了 Python MDO 框架的負載計算邏輯：流體力學模組（如 Vortex Lattice Method, VLM）只需將單個翼段的積分扭矩輸出為一個純量，Python 腳本便能直接將該純量寫入 `*CLOAD` 的資料行中，而無須費時計算如何將扭矩等效分佈為周界上的切線集中力。然而，這是一個極度強烈的運動學拘束，會人為地大幅增加施力端局部的截面剛度（使其不會發生截面翹曲變形），因此在判讀靠近該參考點區域的應力分佈時，應排除應力奇異性（Stress Singularity）造成的局部假象。

### 場景四：局部座標轉換與結構自重之疊加分析

人力飛機的機翼通常被設計為具備顯著的上反角（Dihedral Angle），以提供滾轉軸向的固有氣動穩定性。當機翼與中央機身接合時，其邊界條件必須沿著傾斜的局部座標系施加。同時，結構的自重在長達 30 公尺的展弦比下，會引發不可忽視的初始下垂變形（Droop）。

\*END STEP

在引入 `*TRANSFORM` 之後，對應節點集合內的所有自由度向量空間都被重新定義。這意味著後續針對這些節點施加的 `*BOUNDARY` 或 `*CLOAD` 數值，都是相對於這個傾斜的局部座標系運作，而非全域（Global）直角座標系 \[12, 13\]。另一方面，自重載荷的施加依賴於 `*DLOAD, GRAV`。CalculiX 求解器在掃描到此關鍵字時，會遍歷 `All_Elements` 集合中所有元素的體積與積分點，並讀取先前在 `*MATERIAL` 區塊下定義的 `*DENSITY` 數值，以此建構全域的質量矩陣，最終與重力加速度向量相乘得出體積力負載向量 \[6\]。若 Python MDO 腳本在材料屬性庫中遺漏了密度定義，求解器雖然不會在語法檢查時報錯，但在組裝負載向量時將產生全零矩陣，導致結構在自重分析步中毫無變形。

### 場景五：包含初始應力場的多步漸進分析（Multi-step Perturbation）

在真實的飛行歷程中，結構並非瞬間承受極限負載。機翼首先承受自身的重量（產生初始下垂），隨後在飛機加速滑跑時產生升力（使機翼向上彎曲）。這種包含初始預力（Pre-tension）的狀態，會顯著改變結構的整體剛度，這在非線性力學中被稱為應力剛化現象（Stress Stiffening）。

在 CalculiX 中，多步分析（Multi-step Analysis）的核心邏輯是狀態的繼承 \[14\]。第二個 `*STEP` 會自動繼承第一個分析步結束時的幾何變形狀態、內部應力場以及接觸狀態。為捕捉大變形下的幾何非線性特徵，每一個 `*STEP` 宣告旁都必須伴隨 `NLGEOM` 參數。在第二步中引入的 `*AMPLITUDE` 確保了高強度升力是以平滑的歷程漸進施加。對於 Python MDO 框架而言，多步分析提供了高度的靈活性：程式可以在單一 `.inp` 腳本中，依序模擬起飛、巡航與高Ｇ力機動等多種飛行負載工況（Load Cases），並在後處理階段一次性提取不同工況下的最大變形量以評估設計可行性。

## 自動化框架整合之常見地雷與避險策略

在建構與操作多學科最佳化（MDO）Python 框架時，工程師經常面臨一個嚴峻的挑戰：純文字層面的 `.inp` 檔案寫入雖然簡單，但卻極易忽視有限元素法底層力學邏輯與軟體版本間的相容性差異。許多在商業軟體（如 Abaqus）中視為理所當然的操作邏輯，一旦未經處理直接轉譯給 CalculiX 執行，往往會引發求解發散、矩陣奇異點或物理意義完全錯誤的災難性結果。以下深度剖析與外部網格生成器對接、單位制轉換以及求解器特性相關的五個核心地雷，並提出具體的避險策略。

### 地雷一：Abaqus 相容性錯覺與後挫曲分析之 Riks 求解器缺失

由於 CalculiX 最初的設計宗旨之一是成為相容於 Abaqus 輸入格式的開源替代方案，開發者極容易陷入「語法可無縫轉移」的危險錯覺。實際上，在薄殼結構與挫曲分析這兩個 HPA 研發的核心領域，兩者存在不可忽視的底層差異 \[3\]。

首要原因在於參數資料行的定義格式分歧。在 Abaqus 的環境下，定義一個均質殼元素的 `*SHELL SECTION` 需要在資料行提供兩個具體數值：總厚度以及厚度方向的數值積分點數量（例如採用辛普森積分法則的 5 個積分點）。然而，CalculiX 內部的殼元素整合邏輯不同，它僅接受第一個參數（厚度），第二個欄位必須嚴格留空或略過 \[3, 9\]。如果 Python 腳本直接套用 Abaqus 的寫法輸入兩個數字，CalculiX 的解析器（Parser）將無法正確對應資料型態，導致整個分析任務在初始化階段便因語法錯誤而宣告失敗。同理，在宣告 `*BUCKLE` 特徵值求解時，Abaqus 根據所選取的求解器（Subspace 或是 Lanczos）有截然不同的參數陣列要求；而 CalculiX 則統一要求輸入提取數量、精度、Lanczos 向量數量與迭代上限四個明確的參數 \[2, 3\]。

更為致命的地雷發生在後挫曲（Post-buckling）極限分析階段。在極致輕量化的薄壁結構設計中，主樑在達到初始特徵值挫曲狀態後，經常還具備一定的殘餘承載能力，但同時對製造過程產生的幾何微小缺陷（Imperfection）極度敏感 \[20\]。商業軟體 Abaqus 通常仰賴強大的弧長法（Riks Method）搭配靜態或動態隱式求解器，來順利跨越挫曲點並追蹤負載-位移曲線下降段的非線性路徑。**然而，CalculiX 至今並未實作 Riks 弧長法求解器**。

面對此一重大技術限制，MDO 應對策略必須轉換為實作「多階段微擾疊加法（Perturbation Method）」。Python 框架必須被設計為一個雙階段的迴圈：

1. 第一階段提交線性 `*BUCKLE` 分析，利用 Python 解析產生的 `.frd` 或 `.dat` 檔案，精確提取第一挫曲模態的特徵向量（即各節點的相對位移分佈）。

2. 在 Python 記憶體中，將這個特徵向量陣列乘上一個極小的比例因數（例如局部殼層厚度的 1% 至 5%），作為初始幾何缺陷，並將其疊加回原始的 `.inp` 節點座標中。

3. 第二階段採用更新後帶有微小缺陷的網格，啟動包含 `NLGEOM` 的 `*STATIC` 漸進非線性分析 \[20\]。此一方法能在缺乏 Riks 求解器的限制下，精準捕捉結構隨負載增加而崩塌的極限承載力。

### 地雷二：無單位制（Unitless System）的隱性縮放與矩陣病態災難

CalculiX 的底層程式碼（由 C 與 Fortran 撰寫）完全不具備物理單位的觀念。對於求解器而言，所有的純量數字皆僅是參與矩陣運算的浮點數 \[1, 21\]。這將單位的一致性與對齊責任完全推卸給了使用者或自動化框架的開發者。

對於一般機械零件而言，採用標準國際單位制（SI, 公尺-公斤-秒）通常不會引發顯著的問題。然而，人力飛機是一個尺度極端兩極化的物理系統：機翼的幾何跨度可能高達 30 公尺，但碳纖維主樑的單層管壁厚度可能僅有 0.125 毫米，材料密度亦遠低於一般金屬。如果 Python 框架採用公尺（m）作為基礎長度單位，薄殼的厚度在數值上將呈現為 $0.000125 \text{ m}$。在有限元素法的底層運算中，薄殼的抗彎剛度（Bending Stiffness）與厚度的三次方成正比。當如此微小的浮點數被進行三次方運算並組裝進入全域剛度矩陣時，巨大的長度跨距與極微小的厚度數值將導致剛度矩陣的條件數（Condition Number）嚴重退化，逼近雙精確度浮點數（Double Precision）的截斷誤差（Truncation Error）極限，最終引發求解器發散或給出毫無物理意義的隨機變形結果。

此外，關於結構自重的計算，許多初階工程師經常誤解 `*PHYSICAL CONSTANTS, NEWTON GRAVITY` 的功用，誤以為必須設定此常數才能計算重力 \[6\]。事實上，該指令專門用於模擬星體間的萬有引力行為；常規的工程結構自重必須嚴格依賴 `*DLOAD, GRAV` 指令搭配重力加速度向量來實現。

強烈建議 Python MDO 框架在全局資料結構中全面採用**毫米-公噸-秒 (mm-Ton-s)** 單位系統。下表列出了從 SI 制轉換至此系統的精確對齊規範：

| 物理參數維度 | SI 制表示法 (避免使用) | 推薦系統：Ton-mm-s 制 | MDO 框架 Python 轉換運算式 | 
|---|---|---|---|
| **長度 (Length)** | $m$ (公尺) | $mm$ (毫米) | $L_{mm} = L_{m} \times 1000$ | 
| **質量 (Mass)** | $kg$ (公斤) | $Tonne$ (公噸) | $M_{ton} = M_{kg} \times 10^{-3}$ | 
| **力量 (Force)** | $N$ (牛頓) | $N$ (牛頓) | 不變，因 $1 \text{ kg} \cdot m/s^2 = 1 \text{ Tonne} \cdot mm/s^2$ | 
| **壓力與模數 (Stress/Modulus)** | $Pa$ ($N/m^2$) | $MPa$ ($N/mm^2$) | $E_{MPa} = E_{Pa} \times 10^{-6}$ | 
| **密度 (Density)** | $kg/m^3$ | $Tonne/mm^3$ | $\rho_{ton} = \rho_{kg} \times 10^{-12}$ | 
| **重力加速度 (Gravity, g)** | $9.81 \text{ } m/s^2$ | $9810 \text{ } mm/s^2$ | $g_{mm} = 9.81 \times 1000$ \[5, 6\] | 

透過實施此一單位映射，極小的厚度數值將轉化為工程上易於處理的 $>0.1$ 浮點數等級，楊氏模數則落在數萬至十幾萬的區間，完美避開了剛度矩陣的數值奇異點危機。

### 地雷三：Gmsh 網格對接與 NSET 元素內部擴展的致命對應錯誤

在構建開源 MDO 工作流時，工程師通常採用 Gmsh 作為參數化幾何生成與網格劃分的引擎。然而，當 Gmsh 生成的表面網格檔（如 `.msh` 轉 `.inp`）與 CalculiX 對接時，往往會發生最令人困惑且難以除錯的節點對應崩潰問題 \[15, 22\]。

其根源來自於 CalculiX 極為特殊的底層架構：**CalculiX 內部求解器根本不存在真正的 2D 殼元素或 1D 樑元素** \[23, 24\]。當我們在 `.inp` 檔中宣告使用 `S8R`（八節點二次薄殼元素）或 `M3D4R`（四節點薄膜元素）時，這只是提供給使用者的語法糖。當 CalculiX 啟動執行緒進行資料預處理時，它會讀取 `*SHELL SECTION` 所賦予的厚度數值，然後在記憶體中自動將這些 2D 表面節點沿著法線方向「擴展（Expand）」為真實的 3D 實體元素（例如將 `S8R` 自動膨脹為 `C3D20R` 二十節點六面體元素）\[22\]。

這個內部膨脹機制會帶來毀滅性的連鎖反應。假設 MDO 腳本透過空間座標運算，自行記錄了翼根處 20 個邊界節點的 ID 號碼，並將這些具體的數字 ID 寫入 `.inp` 的 `*BOUNDARY` 區塊中。當 CalculiX 將單層的殼網格膨脹為三維實體時，為了建構上下表面與中立面的連通性，求解器會在內部自動生成成千上萬個全新的「衍生節點（Derived Nodes）」\[22\]。此時，原先腳本指定的節點 ID 很可能已經被重新排列，或者原本計畫拘束整個截面的指令，最終只鎖定在了膨脹後實體模型的最底層表面上，導致結構剛度大幅弱化，產生虛假的巨大位移或計算發散。

為徹底根除此一地雷，MDO 應對策略必須從 Gmsh 的幾何建模階段著手：

1. **嚴格拒絕依賴絕對節點 ID**：在任何 Python 腳本中，絕對不可透過座標距離去尋找並抓取特定的節點號碼寫入約束檔案。

2. **啟動物理群組導出機制**：在 Gmsh 的 `.geo` 建模腳本中，必須將欲施加邊界或負載的幾何面與幾何線宣告為「實體群組（Physical Groups）」，並在選項中強制加入 `Mesh.SaveGroupsOfNodes = 1;` \[25\]。這項指令會迫使 Gmsh 將這些幾何群組內的節點，以純文字命名的 `*NSET`（例如 `NSET=Root_Fix`）完整導出至 `.inp` 檔內 \[8\]。

3. **字串導向的邊界條件賦予**：後續 Python 框架生成 `*BOUNDARY`、`*CLOAD` 或 `*RIGID BODY` 關鍵字時，施加的對象參數只能使用這些字串名稱。CalculiX 在內部執行殼元素擴展時，擁有一套安全繼承機制：它會自動辨識這些名為 `Root_Fix` 的 `*NSET`，並將膨脹過程中產生在對應截面上的所有新節點，全數妥善囊括進這個集合名稱中。如此一來，邊界拘束便能完美且精準地包覆住整個 3D 實體截面。

### 地雷四：複合材料法向量反轉、重疊干涉與偏置（Offset）失效

在複雜的機翼主結構設計中，主樑（Spar）不可避免地需要與縱向排列的翼肋（Ribs）相交與接合。若兩者的幾何模型皆被簡化為殼元素網格以節省運算資源，在網格介面的拓樸處理上極易引發人為的剛度暴增或幾何穿透錯誤 \[26\]。

首先是法向量一致性的問題。對於等向性金屬材料而言，殼元素的法線方向（Normal Vector）朝向內部或外部對分析結果並無影響。但對於宣告為 `COMPOSITE` 的複合材料而言，法向量是決定疊層堆疊順序的絕對基準座標系。如果 Gmsh 在自動佈網格時，因為幾何面法線未對齊而導致生成的網格表面法向量雜亂無章（例如圓管上半部法線朝外，下半部法線朝內），同一個 `[0°/45°/90°]` 的材料配方參數，在結構的下半部會被求解器反轉解讀為 `[90°/45°/0°]`。這將在結構矩陣中產生完全顛倒且不可預期的非對稱耦合剛度矩陣（Asymmetric Coupling Stiffness）。Python 框架必須在網格匯出前，呼叫 Gmsh API（如 `gmsh.model.mesh.reorient` 功能）強制統一所有薄壁面的法線方向。

其次是十字交會處的材料重疊干涉（Material Overlap）。當翼肋以 T 型或十字型交會於主樑時，若沒有給定確切的偏置量，CalculiX 將殼網格往兩側膨脹為實體後，兩塊實體模型在交會處的體積會發生重疊。這等同於求解器在該空間區域計算了兩次質量與剛度，導致局部應力集中假象與整體扭轉剛度異常飆高 \[26\]。在實作中，使用 `*SHELL SECTION, OFFSET=-0.5` 或 `0.5` 來決定厚度膨脹的方向是必要的，但需特別注意，在 `COMPOSITE` 的參數環境下，CalculiX 嚴格禁止使用「節點厚度（NODAL THICKNESS）」來進行動態干涉退讓 \[9\]。因此，治本之道在於 MDO 的幾何建構引擎（Geometry Builder）在畫製 CAD 幾何外型曲線時，就應該考量積層板的物理厚度，提早預留出翼肋與主樑間的幾何間隙（Gap），並利用 `*TIE` 接觸約束將兩者的網格節點動態綁定，藉此徹底杜絕擴展後的實體穿透問題。

### 地雷五：\*EQUATION 與多點約束（MPC）引發的層級衝突與過度拘束

為了精準控制不同網格密度的部件之間的位移傳遞，或是施加特殊的扭轉負載，HPA 工程師大量依賴 `*EQUATION`、`*RIGID BODY` 或 `*TIE` 將節點進行運動學綁定。這些關鍵字在數學底層的運作邏輯，等同於建立多點約束（Multipoint Constraint, MPC）聯立方程式，並從全域系統剛度矩陣中「消除（Eliminate）」等式左側的從屬自由度（Dependent Degree of Freedom）\[11\]。

然而，CalculiX 的稀疏矩陣求解器（如 SPOOLES 或 PARDISO）對於自由度的消除具備極度嚴格的層級規定（Hierarchy Rules）：**任何一個節點的單一自由度，在整個分析任務的所有 `*EQUATION`、`*RIGID BODY`、甚至 `*BOUNDARY` 宣告中，只能被當作一次「從屬節點」** \[11\]。一旦某個自由度被消除，該自由度在矩陣中便不復存在。如果 Python 腳本在自動化分配拘束條件時缺乏防呆機制，不慎將節點 $A$ 指定為某個 `*EQUATION` 的從屬點，同時又將該節點納入受 `*RIGID BODY` 控制的 `NSET` 之中（RBE2 內部同樣視該集合內所有節點為從屬點），求解器將陷入無法解析消除順序的邏輯死胡同，進而拋出過度拘束（Overconstraint）錯誤並立即崩潰中止運算 \[27\]。

此外，在宣告 `*EQUATION` 時，等式項目的排列順序會對矩陣求解的數值穩定性產生直接影響。CalculiX 規定方程式的第一個變數預設為被消除的從屬變數 \[11\]。若該變數在物理現象上的主導係數極小，而在求解過程中被迫除以該微小係數以消除變數時，將引發災難性的捨入誤差。因此，在 Python 腳本動態生成線性約束方程式字串時，務必加入數學條件判斷，確保將權重係數最大（最能主導變形特徵）的自由度排列於 `*EQUATION` 定義行的首位，從而維護整體矩陣求解的穩定性與高效率。

## 多學科最佳化（MDO）整合架構與資料流傳遞

將上述力學機制與地雷避險策略融入 Python 自動化框架後，HPA 的多學科最佳化架構方能展現其實用價值。在一個典型的演算法迴圈（例如採用非線性規劃求解器 SLSQP 或非支配排序遺傳演算法 NSGA-II）中，Python 框架扮演著中樞神經的角色，負責協調幾何生成、空氣動力學計算與 CalculiX 結構求解的資料流傳遞。

在每次迴圈的起始階段，Python 首先產生一組設計變數（Design Variables），這可能包含主樑的根部半徑、末端半徑、碳纖維各疊層角度（$0^\circ, \pm 45^\circ, 90^\circ$）以及層數。這些變數被傳遞給 OpenCASCADE 或 Gmsh Python API 生成 3D 幾何輪廓與網格。隨後，空氣動力學模組（VLM）根據飛行速度與攻角，計算出沿翼展方向的升力與俯仰力矩分佈。Python 必須內建一套插值演算法（如徑向基函數網路 Radial Basis Functions, RBF），將 VLM 輸出在氣動網格中心點的負載，精準且守恆地映射至 CalculiX 生成的結構網格節點上，並動態寫入 `*CLOAD` 指令區塊內。

當 `.inp` 檔案組裝完畢後，Python 利用 `subprocess` 模組呼叫作業系統層級的 `ccx` 執行檔。為了達到 MDO 對運算時間的極限苛求，必須透過設定環境變數 `OMP_NUM_THREADS` 來充分解鎖 CalculiX 的多核心平行運算效能 \[21\]。當 CalculiX 退出運算後，Python 隨即接管後處理任務。它必須略過無法被標準 ASCII 處理的 `.frd` 幾何圖形檔，轉而以高效的文字解析腳本掃描 `.dat` 報告檔。從中提取代表全域剛度與質量的最大翼端下垂變形量與結構總重，並從元素應力輸出區塊讀取各積分點的應力張量，藉以計算蔡-吳複合材料破壞指標（Tsai-Wu Failure Index）。這些數值最終被封裝為 MDO 的目標函數（Objective Function，如重量最小化）與非線性約束條件（Non-linear Constraints，如變形量與破壞指標不得超標），回傳給最佳化演算法進行下一代參數的演化。

## 結論

建構一套專注於人力飛機（HPA）設計且基於 Python 與 CalculiX 2.22 的多學科最佳化（MDO）結構求解器，不僅需要深厚的程式開發能力，更是一項高度依賴非線性有限元素運算邏輯與複合材料底層物理機制的系統工程。因為 CalculiX 先天具備將薄殼元素隱性擴展為三維實體的獨特軟體架構，開發團隊絕對不能天真地將其視為一個輸入數據便能獲得精確答案的「黑盒子」求解器。

要確保分析管線（Pipeline）在處理極致輕量化薄壁結構時具備絕對的強健性與自動化容錯率，開發框架必須嚴格遵守以下三大核心防呆準則：第一，強制摒棄依賴絕對節點 ID 的操作惡習，全面改以具備物理群組屬性的字串名稱（NSET/ELSET）作為網格幾何與約束條件溝通的唯一介面；第二，在 Python 資料生成層級實施滴水不漏的 **Ton-mm-s** 單位制轉換，防範極薄壁結構導致的矩陣數值奇異點崩潰；第三，針對 Abaqus 語法相容性的迷思進行解構，放棄尋找內建的 Riks 後挫曲分析捷徑，轉而透過 Python 腳本自主實作特徵向量提取與微小缺陷疊加的微擾分析迴圈。唯有深刻理解並駕馭上述 `.inp` 語法的數學意義與底層矩陣運作限制，方能徹底釋放 CalculiX 作為開源結構求解器在平行運算與參數化最佳化上的強大潛力，為 HPA 的結構設計開拓出具備實用價值與理論深度的最佳幾何與疊層方案。
